Erkenntnisse aus der Analyse

Meine bisherigen Fähigkeiten im Bereich der Single Page Application Entwicklung und der Umsetzung von klassischen Client-Server Modellen hat mich zunächst beim Arbeiten mit Kirby nicht weiter gebracht. Zudem ist der Umgang mit einem Flat-File-System neuartig, wenn man im Alltag die Vorzüge von Relationen und Datenbanksystemen einsetzt. Die Umsetzung eines relationalen Datenmodells gestaltet sich eher schwierig und die Speicherung von Daten in Textdateien erscheint zunächst befremdlich. Dennoch bietet Kirby mit Blick auf die Auslieferung durchaus Vorteile.

In den bereitgestellten Tutorials der Kirby-Entwickler und der Community wird das Thema der Frontend-Entwicklung eher minimal gehalten. Im Laufe des Semesters hat sich herauskristallisiert, dass die Optimierung des Frontends gesondert notwendig ist. Das war ein Lernprozess im Umgang mit Kirby, der dazu geführt hat in der zweiten Version für die Abgabe Webpack und Node.js zu verwenden um die benötigten Ressourcen besser zu verarbeiten.

Zudem gestaltet sich die Arbeit mit der Kirby-Api ausserhalb des Panel-Kosmos eher schwierig. Für Requests steht sessionbasierte Authentifizierung und Basic-Authentication bereit. Die Speicherung von Zugangsdaten in Form einer Benutzername-Passwort-Kombination im Frontend kommt aus meiner Sicht nicht in Frage (Security). Deshalb mussten Requests für die Suche mit Content-Represenationen und virtuellen Seiten abgebildet werden um die Suchinhalte über eine API abrufen zu können. Diese Requests konnten wiederum durch diverse Header und über Cross-Origin Einstellungen gegen Fremdzugriffe einigermaßen abgesichert werden. Ein Abruf der API über JSON-Web Token oder generell tokenbasierter Authentifizierungsmethoden wäre wünschenswert gewesen.

Verbesserungen am System

Neuaufbau mit Kirby 3

Im Rahmen der Winter Werkschau 19/20 wurde das System komplett neu aufgebaut und mit Kirby 3 umgesetzt.

Datenspeicherung und Bearbeitung

Für die Bedienung des Systems war es mir besonders wichtig, Daten über das Panel von Kirby verwalten zu können. Hierbei insbesondere Daten, welche nicht von Studenten erfasst werden, sondern für die Benutzung des Systems zur Verfügung stehen müssen. Beispiel: Optionen eines Dropdowns, Dozenten, Adressen ect.

Als Datenspeicher für zentrale Daten dient die site.txt von Kirby. Hierin werden alle global zur Verfügung stehenden Daten gespeichert und können von Administratoren über das Panel gepflegt werden. Somit entfällt die Bearbeitung im Code und macht das Anpassen von Inhalten deutlich leichter.

Trennung von Administratoren und Studenten

Im bisherigen System gab es keine Trennung von Administratoren und Studenten. Dieser Umstand sollte geändert werden um zum Einen eine verbesserte Übersichtlichkeit zu bieten, zum Anderen um den Nutzern nur die Informationen und Bearbeitungsmöglichkeiten zu geben, welche für die jeweilige Rolle vorgesehen sind.

Um eine Separierung zu erreichen wurde die Funktionalität für die unterschiedlichen Panels in ein Plugin ausgelagert um die Wartbarkeit und Übersichtlichkeit zu erhalten.

Aufteilung der Funktionalität in Plugins

Vorausschauend auf den kommenden Funktionsumfang wurden Funktionalitäten in Plugins unterteilt. Wie bereits am Beispiel des Role Panel aufgezeigt, bietet die aktuelle Fassung Plugins für die folgenden Bereiche.

  • Studenten / Abschlussarbeiten / Semesterarbeiten / Teamarbeiten
  • Speaker und Talks
  • Suche
  • Infrastruktur (in Entwicklung)
  • Block Plugin (für einen eigenentwickelten Page-Builder)

Frontendentwicklung mit Webpack und Node.js

Um auch moderne Webentwicklung mit Webpack abbilden zu können wurde die Entwicklung des Frontends ausgelagert.

Features der Abgabe:

  1. leichtere Verwaltung der Abhängigkeiten (Frontend Module)
  1. Vorentwicklung der HTML / SCCS - CSS sowie JS Struktur und Erstellung eines reinen HTML-Templates
  1. Zusammenfassung und Verkleinerung benötigter Ressourcen (JS und CSS)
  1. Webpack Config für die Erstellung der relevanten Kirby-Struktur (assets Verzeichnis)
